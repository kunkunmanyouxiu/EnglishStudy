# 本地AI默写背单词APP开发文档（Gemma 4 E4B + SQLite 离线版）

# 1. 文档概述

## 1.1 文档目的

本文档为本地AI默写背单词APP的完整开发规范，明确项目架构、技术选型、模块功能、开发流程及接口规范，用于指导AI工具生成可直接编译运行的代码，确保开发过程标准化、高效化，最终实现“离线运行、本地AI判题、数据隐私安全”的核心需求，新增自定义单词上传（支持纸质单词录入）及上传前AI预判断功能，适配用户个性化词库需求。

## 1.2 项目背景

市面现有背单词APP均以“选择题认词”为核心，无法实现“手动输入释义/单词”的默写场景，且多依赖云端服务器，存在数据隐私泄露风险；同时缺乏自定义单词上传功能，无法满足用户录入纸质版单词、创建专属词库的需求。本项目旨在开发一款纯本地运行的背单词APP，内置Gemma 4 E4B大模型实现AI语义判题，采用SQLite本地数据库存储数据，全程不联网、不上传任何用户数据，解决“眼熟手废”、隐私安全及自定义单词录入三大痛点，支持用户上传纸质版单词及释义，上传前通过AI预判断确保单词与释义匹配度，提升自定义词库质量。

## 1.3 核心目标

- 功能目标：实现“英文→中文默写”“中文→英文默写”两种核心场景，AI语义判题（支持同义词、近义词判定），艾宾浩斯复习算法，本地词库管理，新增自定义单词上传（纸质版单词录入）、上传前AI预判断、专属词库创建功能。

- 技术目标：Gemma 4 E4B模型本地集成（离线运行），SQLite本地数据存储，Android端流畅运行（判题响应≤1秒），无任何云端依赖，实现自定义单词上传逻辑与AI预判断功能，确保上传数据合规可用。

- 隐私目标：所有用户数据（词库、学习记录、默写答案、自定义上传单词）100%存储在本地，不申请网络权限，不发送任何数据至云端。

## 1.4 适用范围

本文档适用于本APP的全流程开发，包括Android端开发、本地大模型集成、SQLite数据库开发、AI判题逻辑开发、自定义单词上传及AI预判断功能开发，供AI工具生成代码、开发人员调试优化使用。

# 3. 系统架构设计

## 3.1 整体架构（纯本地，无云端）

本APP采用三层架构，全程本地运行，无任何云端交互，架构如下：

1. 表现层（UI层）：Android原生页面，负责用户交互（选词书、默写输入、结果展示、学习记录查看、自定义单词上传、纸质单词拍照识别等）。

2. 业务逻辑层：核心业务处理，包括艾宾浩斯复习算法、AI判题逻辑、词库管理、数据库操作、模型加载与调用、自定义单词上传逻辑、上传前AI预判断逻辑、纸质单词识别逻辑。

3. 数据层：SQLite本地数据库（存储词库、学习记录、错题本、用户自定义上传单词）+ Gemma 4 E4B本地模型（负责AI判题、上传前预判断）。

架构流程图：表现层 → 业务逻辑层 → 数据层（SQLite + 本地模型），所有交互均在本地完成，无网络请求。

## 3.2 核心模块划分

APP分为5个核心模块，新增自定义单词上传相关功能至对应模块，各模块独立开发、相互调用，具体如下：

- 模块1：本地模型集成模块（负责Gemma 4 E4B模型加载、Prompt调用、判题结果返回、自定义单词上传前AI预判断）

- 模块2：SQLite数据库模块（负责词库、学习记录、错题本、自定义上传单词的增删改查，支持专属词库创建）

- 模块3：核心默写模块（负责两种默写模式的页面展示、用户输入、判题触发）

- 模块4：复习算法模块（负责艾宾浩斯遗忘曲线计算、复习计划生成、错题复盘）

- 模块5：辅助功能模块（负责语音朗读、词库导入、UI交互优化、自定义单词上传、纸质单词拍照识别）

# 4. 详细模块开发规范（AI工具生成代码核心依据）

## 4.1 模块1：本地模型集成模块（核心优先级最高）

### 4.1.1 模块功能

实现Gemma 4 E4B模型的本地加载、初始化，接收业务层传递的“单词、标准释义、用户输入”，调用模型进行语义判题，返回“正确/部分正确/错误”及标准释义提示；新增自定义单词上传前AI预判断功能，接收用户上传的“单词、自定义释义”，判断两者语义匹配度，返回预判断结果及优化建议，确保上传单词与释义的合理性。

### 4.1.2 技术实现规范

1. 模型准备：

2. 下载Gemma 4 E4B 4-bit量化模型（格式：*.gguf 或 *.bin），放入Android项目的assets目录下。

3. APP第一次启动时，将assets目录下的模型文件复制到手机本地存储（避免assets目录只读限制），后续启动直接加载本地存储的模型。

4. MediaPipe LLM集成：

5. 在Android项目的build.gradle中添加MediaPipe LLM依赖（具体依赖代码见附录）。

6. 创建ModelManager工具类，封装模型加载、初始化、调用方法，确保模型加载仅执行一次（避免重复加载占用内存）；新增预判断方法，专门处理自定义单词上传前的语义匹配判断。

7. 模型加载失败时，弹出提示“模型加载失败，请重启APP”，并记录错误日志（本地存储日志，不上传）；预判断失败时，提示“预判断失败，请检查单词和释义后重试”。

8. AI判题逻辑：

9. 接收业务层传递的三个参数：word（标准单词）、definition（标准释义）、userInput（用户输入答案）。

10. 使用固定Prompt（见4.1.3）格式化参数，传递给Gemma 4 E4B模型。

11. 解析模型返回结果，提取“正确/部分正确/错误”及标准释义，返回给业务层。

12. 判题响应时间控制在1秒内，若超过1秒，弹出提示“判题中，请稍候”。

13. 自定义单词上传前AI预判断逻辑（新增）：

14. 接收业务层传递的两个参数：customWord（用户上传的单词）、customDefinition（用户上传的单词释义）。

15. 使用固定预判断Prompt（见4.1.4）格式化参数，传递给Gemma 4 E4B模型，重点判断单词与释义的语义匹配度。

16. 解析模型返回结果，提取“匹配度等级”（高/中/低）、“匹配说明”及“优化建议”，返回给业务层。

17. 预判断响应时间控制在1秒内，若超过1秒，弹出提示“预判断中，请稍候”；预判断匹配度为“低”时，提示用户检查单词与释义的准确性，可选择修改后重新预判断或强制上传。

### 4.1.3 离线AI判题Prompt（固定不变，直接调用）

```text
你是一个严格的英语单词判题老师，仅负责判断用户输入的答案与标准释义是否语义一致，不添加任何多余解释，严格遵循以下规则：
任务：判断用户输入的中文意思（或英文单词），是否与标准释义（或标准单词）语义一致。
标准单词：{{word}}
标准释义：{{definition}}
用户输入：{{user_input}}
判题规则：
1.  若为英文→中文默写：用户输入的中文是标准释义的同义词、近义词、口语化简写，均判定为【正确】；仅写出部分核心含义，判定为【部分正确】；完全无关、语义相反，判定为【错误】。
2.  若为中文→英文默写：用户输入的英文单词拼写正确（允许1-2个字母手误）、语义与标准单词一致，判定为【正确】；拼写错误但语义接近，判定为【部分正确】；拼写错误且语义无关，判定为【错误】。
输出格式（严格遵守，不得添加任何多余内容）：
结果：正确/部分正确/错误
标准释义：{{definition}}
补充提示：（仅当部分正确/错误时添加，正确时不写）
```

### 4.1.4 自定义单词上传前AI预判断Prompt（新增，固定不变）

```text
你是一个专业的英语单词语义匹配审核员，仅负责判断用户上传的单词与释义的语义匹配度，不添加任何多余解释，严格遵循以下规则：
任务：判断用户上传的英文单词与中文释义是否语义匹配，给出匹配度等级、匹配说明及优化建议。
用户上传单词：{{custom_word}}
用户上传释义：{{custom_definition}}
判断规则：
1.  匹配度等级分为“高”“中”“低”三级：
    - 高：单词与释义完全匹配，释义准确覆盖单词核心含义，无歧义；
    - 中：单词与释义基本匹配，释义大致符合单词含义，但存在表述不精准、遗漏次要含义的情况；
    - 低：单词与释义完全不匹配，语义无关或相反，无法通过释义理解单词含义。
2.  匹配说明需简要说明匹配情况，不超过50字；优化建议仅在匹配度为“中”“低”时给出，简洁实用，不超过30字。
输出格式（严格遵守，不得添加任何多余内容）：
匹配度等级：高/中/低
匹配说明：{{match_explanation}}
优化建议：{{optimization_suggestion}}（匹配度为高时不写）
```

## 4.2 模块2：SQLite数据库模块

### 4.2.1 模块功能

负责本地词库管理、学习记录存储、错题本管理、用户自定义上传单词存储，提供增删改查接口，供其他模块调用；支持用户创建专属词库，将自定义上传的单词归类存储，所有数据均存储在本地SQLite数据库，不进行任何云端同步。

### 4.2.2 数据库表结构（SQL代码直接执行，新增自定义相关字段）

```sql
-- 1. 单词表（存储所有词库数据，新增自定义标识字段）
CREATE TABLE IF NOT EXISTS words (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    word TEXT NOT NULL, -- 英文单词
    phonetic TEXT, -- 音标（可选）
    definition TEXT NOT NULL, -- 标准释义（用于AI判题）
    example TEXT, -- 例句（可选）
    word_type TEXT, -- 单词类型（如考研、四级、六级、自定义）
    is_collect INTEGER DEFAULT 0, -- 是否收藏（0=未收藏，1=收藏）
    is_custom INTEGER DEFAULT 0, -- 是否为自定义上传单词（0=系统词库，1=自定义）
    custom_book_id INTEGER DEFAULT 0, -- 关联专属词库ID（0表示未归类，自定义单词需关联对应词库）
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP -- 新增时间
);

-- 2. 学习记录表（用于艾宾浩斯复习，无修改）
CREATE TABLE IF NOT EXISTS study_records (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    word_id INTEGER NOT NULL, -- 关联单词表的id
    status INTEGER NOT NULL DEFAULT 0, -- 学习状态（0=不会，1=模糊，2=掌握）
    review_count INTEGER DEFAULT 0, -- 复习次数
    last_review_time TIMESTAMP DEFAULT CURRENT_TIMESTAMP, -- 最后复习时间
    next_review_time TIMESTAMP NOT NULL, -- 下次复习时间（艾宾浩斯计算得出）
    FOREIGN KEY (word_id) REFERENCES words(id) ON DELETE CASCADE -- 级联删除
);

-- 3. 错题本（存储用户默写错误的单词，无修改）
CREATE TABLE IF NOT EXISTS error_words (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    word_id INTEGER NOT NULL, -- 关联单词表的id
    error_count INTEGER DEFAULT 1, -- 错误次数
    last_error_time TIMESTAMP DEFAULT CURRENT_TIMESTAMP, -- 最后错误时间
    error_answer TEXT, -- 最后一次错误答案
    FOREIGN KEY (word_id) REFERENCES words(id) ON DELETE CASCADE
);

-- 4. 词库分类表（用于区分不同类型的词库，新增自定义词库标识）
CREATE TABLE IF NOT EXISTS word_category (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    category_name TEXT NOT NULL, -- 词库名称（如考研英一、四级、自定义词库1）
    word_count INTEGER DEFAULT 0, -- 该词库的单词数量
    is_custom INTEGER DEFAULT 0 -- 是否为用户自定义词库（0=系统词库，1=自定义）
);

-- 5. 自定义词库关联表（新增，用于管理用户创建的专属词库与单词的关联）
CREATE TABLE IF NOT EXISTS custom_word_book (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    book_name TEXT NOT NULL, -- 自定义词库名称（如“纸质单词本1”“考研高频自定义”）
    create_time TIMESTAMP DEFAULT CURRENT_TIMESTAMP, -- 词库创建时间
    word_count INTEGER DEFAULT 0, -- 词库内单词数量
    is_delete INTEGER DEFAULT 0 -- 是否删除（0=未删除，1=已删除，逻辑删除）
);
```

## 4.3 模块3：核心默写模块（用户交互核心）

### 4.3.1 模块功能

实现两种核心默写场景，负责用户输入、判题触发、结果展示，是APP的核心交互页面，要求UI简洁、操作流畅，突出“默写输入”的核心体验；支持选择自定义词库进行默写，与系统词库默写逻辑一致。

### 4.3.2 页面设计规范（新增自定义词库选择相关）

1. 页面组成（两种模式共用一个页面，切换模式即可，新增自定义词库选择入口）：

2. 顶部：模式切换按钮（英文→中文 / 中文→英文）、词库选择按钮（可选择系统词库或自定义词库）、当前词库名称、当前单词序号/总数量。

6. UI要求：

7. 简洁极简，无多余动画，重点突出输入框和核心按钮；词库选择按钮明显，点击后弹出弹窗，区分系统词库和自定义词库。

8. 输入框放大，字体适中，支持输入纠错（如中文拼音联想、英文拼写提示）。

9. 颜色搭配：背景为浅色（白色/浅灰色），输入框边框清晰，判题结果颜色区分明显，避免刺眼。

### 4.3.3 业务逻辑规范（新增自定义词库默写逻辑）

1. 模式切换：用户点击模式按钮，切换两种默写模式，页面实时刷新展示内容（无需重启页面）。

2. 单词加载：进入默写页面时，用户选择词库（系统/自定义），从SQLite数据库加载对应词库的单词（自定义词库加载custom_book_id匹配的单词），按随机顺序排列（避免固定顺序记忆）。

3. 用户输入：输入框支持中文、英文输入，自动剔除空格、标点符号（避免因格式问题导致判题错误）。

4. 判题触发：点击提交按钮后，禁用输入框和提交按钮，调用本地模型集成模块进行判题，1秒内返回结果并展示。

5. 结果处理：

6. 判题正确：更新学习记录状态为“掌握”，不添加到错题本，3秒后自动加载下一个单词（自定义单词与系统单词处理逻辑一致）。

7. 判题部分正确：更新学习记录状态为“模糊”，不添加到错题本，展示补充提示，3秒后加载下一个单词。

8. 判题错误：更新学习记录状态为“不会”，添加到错题本（记录错误答案），展示标准释义和补充提示，用户可点击“重新默写”或“跳过”。

## 4.4 模块4：复习算法模块（艾宾浩斯离线版）

### 4.4.1 模块功能

基于艾宾浩斯遗忘曲线，根据用户的学习状态（不会/模糊/掌握），计算每个单词的下次复习时间，生成每日复习计划，确保用户高效记忆单词；支持自定义上传单词的复习计划生成，与系统单词复习逻辑一致，不做额外修改。

### 4.4.2 艾宾浩斯算法规范（本地离线计算，无修改）

1. 核心遗忘曲线时间节点（基于单词学习状态）：

2. 状态为“不会”（首次错误/完全不会）：下次复习时间 = 当前时间 + 1小时、1天、3天、7天、14天、30天。

3. 状态为“模糊”（部分正确）：下次复习时间 = 当前时间 + 12小时、2天、5天、10天、20天。

4. 状态为“掌握”（完全正确）：下次复习时间 = 当前时间 + 7天、14天、30天、60天。

5. 算法逻辑：

6. 每次判题后，根据结果更新单词的学习状态，调用算法计算下次复习时间，更新到study_records表（自定义单词与系统单词统一处理）。

7. 每日启动APP时，自动查询study_records表，筛选出“next_review_time ≤ 当前时间”的单词，作为今日复习单词（包含自定义上传单词）。

8. 复习时，若单词再次判题正确，提升学习状态（如“模糊”→“掌握”），重新计算下次复习时间；若错误，降低学习状态（如“掌握”→“模糊”），缩短下次复习时间。

## 4.5 模块5：辅助功能模块（重点新增自定义单词上传相关功能）

### 4.5.1 模块功能

提供语音朗读、词库导入、收藏管理、学习统计等辅助功能，提升用户体验，不影响核心默写功能；新增自定义单词上传功能（支持手动输入、纸质单词拍照识别录入）、上传前AI预判断、自定义词库创建与管理功能，满足用户录入纸质版单词、创建专属词库的需求。

### 4.5.2 具体功能实现规范（新增自定义单词上传相关功能）

1. 本地语音朗读（无修改）：

2. 调用Android原生TTS，支持英文单词、音标、例句的英美发音切换。

3. 点击发音按钮时，朗读当前展示的内容（英文→中文模式朗读英文，中文→英文模式朗读中文释义）。

4. 支持调节朗读音量、语速，默认语速适中，音量跟随系统。

5. 词库导入（无修改）：

6. 支持用户通过Excel/TXT文件导入自定义词库，导入格式：单词|音标|标准释义|例句|词库类型。

7. 导入时，自动校验格式，格式错误的单词跳过，提示导入成功数量和失败数量。

8. 导入的单词自动存入words表和对应的word_category表。

9. 学习统计（新增自定义单词统计）：

10. 统计今日默写单词数、正确数、错误数、复习单词数（区分系统单词和自定义单词）。

11. 统计累计学习单词数、累计复习次数、错题数（包含自定义上传单词）。

12. 统计自定义词库数量、自定义单词总数，以图表（折线图/柱状图）展示近7天的学习数据（本地绘制，不依赖第三方图表库）。

13. 收藏管理（无修改）：

14. 用户可在默写页面、单词详情页点击收藏按钮，标记单词为收藏（包含自定义上传单词）。

15. 提供收藏单词列表，支持批量删除、批量复习。

16. 自定义词库创建与管理（新增）：

17. 用户可在“我的词库”页面，点击“创建自定义词库”，输入词库名称（如“纸质单词本”“每日新增”），完成创建，系统自动生成词库ID，存入custom_word_book表。

18. 支持修改自定义词库名称、逻辑删除自定义词库（删除后不显示，但词库内单词仍保留，可重新关联其他词库）。

19. 展示所有自定义词库，显示词库名称、单词数量、创建时间，支持点击进入词库，查看词库内所有自定义单词，可进行单词删除、修改操作。

20. 自定义单词上传（核心新增，支持纸质单词录入）：

21. 上传方式：支持两种方式，用户可自由选择：
          

    - 手动输入：用户手动填写单词、音标（可选）、释义、例句（可选），选择关联的自定义词库。

    - 拍照识别（可选）：用户点击“拍照录入”，调用手机相机拍摄纸质单词（需提前申请相机权限，用户可拒绝），通过本地OCR识别提取单词和释义，自动填充到输入框，用户可手动修改纠正后提交。

22. 上传前AI预判断：用户填写/识别完成后，点击“预判断”按钮，调用模块1的预判断功能，传入单词和释义，获取匹配度结果；
          

    - 匹配度“高”：提示用户“匹配度高，可直接上传”，支持点击“立即上传”。

    - 匹配度“中”：提示用户“匹配度中等，建议优化释义”，展示优化建议，用户可修改后重新预判断，或直接点击“忽略建议，上传”。

    - 匹配度“低”：提示用户“匹配度低，单词与释义不匹配”，展示优化建议，用户需修改后重新预判断，不可直接上传（避免无效单词录入）。

23. 上传逻辑：预判断通过后，点击“上传”，将单词信息（标记is_custom=1，关联对应的custom_book_id）存入words表，同时更新custom_word_book表的word_count字段（自动+1），提示“上传成功”。

24. 批量上传：支持批量手动输入或批量拍照识别（最多一次上传10条），批量进行AI预判断，批量上传（仅允许匹配度“高”“中”的单词批量上传，“低”的需单独修改）。

# 5. 开发流程（AI工具生成代码顺序，新增自定义相关开发步骤）

为确保开发顺利，避免模块依赖冲突，AI工具生成代码需按以下顺序执行，每完成一个模块，进行调试，确认无问题后再进行下一个模块：

1. 第一步：搭建Android项目基础框架（配置Kotlin环境、添加MediaPipe LLM依赖、权限配置，新增相机权限配置（可选））。

2. 第二步：开发SQLite数据库模块（创建数据库、实现核心增删改查接口、导入测试词库，新增自定义词库相关表及接口）。

3. 第三步：开发本地模型集成模块（加载Gemma 4 E4B模型、实现AI判题逻辑、调试判题响应速度，新增上传前AI预判断方法及Prompt调用）。

4. 第四步：开发核心默写模块（实现页面UI、用户输入、判题触发、结果展示，对接模型和数据库模块，新增自定义词库选择功能）。

5. 第五步：开发复习算法模块（实现艾宾浩斯算法、对接学习记录接口、生成复习计划，兼容自定义单词复习）。

6. 第六步：开发辅助功能模块（语音朗读、词库导入、学习统计、收藏管理，新增自定义词库创建与管理、自定义单词上传、纸质单词拍照识别功能）。

7. 第七步：整体调试（优化判题速度、修复UIbug、测试离线运行稳定性、验证数据隐私安全性，重点调试自定义单词上传、AI预判断、自定义词库管理功能）。

# 6. 关键注意事项（AI生成代码需规避的问题，新增自定义相关注意事项）

1. 模型加载：确保模型仅加载一次，避免重复加载导致内存溢出；模型复制到本地存储时，需判断是否已存在，避免重复复制浪费时间。

2. 数据隐私：全程不申请网络权限，不写任何网络请求代码；所有数据均存储在SQLite数据库，不写入外部存储的公共目录；自定义上传的单词数据与系统数据分开标记，确保数据安全。

3. 判题精度：严格使用固定Prompt，解析模型返回结果时，需处理异常情况（如模型返回格式错误），避免APP崩溃；上传前预判断需严格遵循Prompt格式，确保匹配度判断准确。

4. 性能优化：控制APP内存占用，模型运行时关闭不必要的后台进程；SQLite操作使用异步线程，避免阻塞UI，导致页面卡顿；批量上传自定义单词时，需分批次处理，避免一次性处理过多数据导致卡顿。

5. 兼容性：适配不同屏幕尺寸的Android手机，输入框、按钮位置自适应；支持Android 8.0及以上版本，避免使用高版本API导致兼容性问题；相机识别功能需适配不同手机的相机权限，用户拒绝权限后，仍可使用手动输入方式上传。

6. 自定义上传相关：拍照识别后需提供手动修改入口，避免识别错误导致无效单词上传；预判断逻辑需清晰，匹配度“低”时禁止直接上传，确保自定义词库质量；自定义词库删除采用逻辑删除，避免误删导致用户数据丢失。
